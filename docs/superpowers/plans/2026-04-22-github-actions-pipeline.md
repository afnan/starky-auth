# GitHub Actions CI/CD Pipeline — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire up a working GitHub Actions pipeline that builds a custom Keycloak image (with Tailwind-built themes) on push to `main`, pushes to `starkyacr`, and deploys to `vm-starky-auth-prod` via `az vm run-command invoke`.

**Architecture:** Two-stage Dockerfile (themes baked at build-time, `kc.sh build` produces an optimized runtime image). GitHub Actions runs on GitHub-hosted Ubuntu runners, authenticates to Azure via OIDC federation on the existing `github-starky-frontend` App Registration (Contributor-on-RG covers all needed permissions). Postgres + Caddy containers on the VM are untouched on deploys — only the Keycloak container swaps image.

**Tech Stack:** Docker, Docker Compose, Keycloak 26.1, Tailwind CSS v3, GitHub Actions, Azure CLI, ACR.

**Prerequisites (already done, per portal runbook Phases 1–6):**
- VM `vm-starky-auth-prod` running at `20.167.84.219` in `rg-starkyapp`
- UAMI `id-starky-auth-vm` attached to VM with `AcrPull` on `starkyacr` and `Storage Blob Data Contributor` on `ststarkyauthbackups`
- App Reg `github-starky-frontend` has Contributor on RG + federated credential `starky-auth-main` for `repo:afnan/starky-auth:ref:refs/heads/main`
- Three repo secrets set: `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`
- DNS A records for `auth.starkyapp.com` and `auth-admin.starkyapp.com` point to the static IP

**One more repo secret to add before Task 7** (instructions in Task 6):
- `VM_ENV_FILE` — multi-line secret containing the entire `.env` contents for the VM (domains, admin creds, DB creds, IP allow-list, etc.) **except** the `IMAGE=` line (which the pipeline sets to the current git SHA per deploy).

**Out of scope for this plan (future work):**
- `backup.sh` azcopy upgrade (pg_dump to blob storage)
- `setup.sh` overhaul (currently assumes Ubuntu + UFW; VM uses NSG)
- `rollback.yml` and `major-upgrade.yml` workflows
- `README.md` rebrand
- Theme coverage beyond `login.ftl` (MFA, forgot-password, etc.)

- [ ] **Step 2.1: Update `caddy/Caddyfile`**

Replace the entire content of `caddy/Caddyfile` with:

```caddyfile
{
    email {$ACME_EMAIL}
}

# ============================================================
# Public authentication domain
# Redirects / to the product site. Proxies realm endpoints to Keycloak.
# Admin console is BLOCKED here — use the admin subdomain.
# Master realm is BLOCKED here — not needed for end-user auth.
# ============================================================
{$AUTH_HOSTNAME} {
    encode gzip

    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
        Permissions-Policy "camera=(), microphone=(), geolocation=()"
        Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; img-src 'self' data: https: blob:; font-src 'self' data: https://fonts.gstatic.com; frame-src 'self'; frame-ancestors 'self'; form-action 'self' https:;"
        -Server
    }

    # Caddy-local health endpoint (doesn't hit Keycloak)
    @health path /health
    handle @health {
        respond "healthy" 200
    }

    # Root → product site
    @root path /
    redir @root https://starkyapp.com 302

    # Block admin console on the public domain
    @admin path /admin /admin/*
    handle @admin {
        respond "Admin access is not permitted on this domain. Use the admin subdomain." 403
    }

    # Block master realm on the public domain (end users live in other realms)
    @master path /realms/master /realms/master/*
    handle @master {
        respond "Not found" 404
    }

    # Everything else → Keycloak
    handle {
        reverse_proxy keycloak:8080
    }
}

# ============================================================
# Admin console — IP-restricted
# ADMIN_ALLOWED_IPS is a space-separated list of IPs/CIDRs.
# Default (127.0.0.1/32) denies every real client.
# ============================================================
{$ADMIN_HOSTNAME} {
    encode gzip

    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains"
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
        -Server
    }

    @allowed remote_ip {$ADMIN_ALLOWED_IPS}
    handle @allowed {
        reverse_proxy keycloak:8080
    }

    handle {
        respond "Access denied. Your IP is not in the admin allow-list." 403
    }
}
```

Create `Dockerfile` at repo root with:

```dockerfile
# Two-stage Keycloak build — themes + optimized runtime.
# See https://www.keycloak.org/server/containers for the recommended pattern.

FROM quay.io/keycloak/keycloak:26.1 AS builder

# Build-time configuration. These cannot be changed at runtime with --optimized.
ENV KC_DB=postgres
ENV KC_FEATURES=token-exchange,admin-fine-grained-authz
ENV KC_HEALTH_ENABLED=true
ENV KC_METRICS_ENABLED=true
ENV KC_CACHE=local

# Copy only the production theme directory (not the Tailwind source).
# The Actions pipeline builds styles.css into themes/starky/login/resources/css/
# BEFORE this Docker build runs, so the output lands here too.
COPY themes/starky/login/ /opt/keycloak/themes/starky/login/

RUN /opt/keycloak/bin/kc.sh build

# Runtime stage — slimmer layer with the augmented distribution only.
FROM quay.io/keycloak/keycloak:26.1
COPY --from=builder /opt/keycloak/ /opt/keycloak/

ENTRYPOINT ["/opt/keycloak/bin/kc.sh", "start", "--optimized"]
```

**Goal:** Push-to-main pipeline: build Tailwind → build Docker image → push to ACR → deploy to VM → wait for health.

**Files:**
- Create: `.github/workflows/build-and-deploy.yml`

- [ ] **Step 5.1: Create the workflow directory**

```bash

```

- [ ] **Step 5.2: Create `.github/workflows/build-and-deploy.yml`**

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]
  workflow_dispatch: {}

concurrency:
  group: starky-auth-deploy
  cancel-in-progress: false

permissions:
  id-token: write  # required for Azure OIDC federation
  contents: read

env:
  IMAGE_REPO: starkyacr.azurecr.io/starky-auth
  RESOURCE_GROUP: rg-starkyapp
  VM_NAME: vm-starky-auth-prod
  VM_APP_DIR: /opt/starky-auth

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: themes/starky/package-lock.json

      - name: Install theme dependencies
        working-directory: themes/starky
        run: npm ci

      - name: Build Tailwind CSS
        working-directory: themes/starky
        run: npm run build:css

      - name: Verify built CSS exists
        run: |
          test -s themes/starky/login/resources/css/styles.css
          echo "Built CSS size: $(wc -c < themes/starky/login/resources/css/styles.css) bytes"

      - name: Azure login (OIDC)
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: ACR login
        run: az acr login --name starkyacr

      - name: Docker build
        run: |
          docker build \
            --tag "${IMAGE_REPO}:${{ github.sha }}" \
            --tag "${IMAGE_REPO}:latest" \
            .

      - name: Docker push
        run: |
          docker push "${IMAGE_REPO}:${{ github.sha }}"
          docker push "${IMAGE_REPO}:latest"

      - name: Sync config + env to VM
        env:
          VM_ENV_FILE: ${{ secrets.VM_ENV_FILE }}
        run: |
          NEW_IMAGE="${IMAGE_REPO}:${{ github.sha }}"

          # Compose the full .env content: secret body + IMAGE line.
          # printf preserves the secret's $-literals (no re-expansion).
          printf '%s\n' "$VM_ENV_FILE" > /tmp/env_body
          printf 'IMAGE=%s\n' "${NEW_IMAGE}" >> /tmp/env_body
          ENV_B64=$(base64 -w0 < /tmp/env_body)

          # The repo's compose + Caddy config sync to the VM on every deploy,
          # keeping repo and VM lockstep — no drift.
          COMPOSE_B64=$(base64 -w0 < docker-compose.yml)
          CADDY_B64=$(base64 -w0 < caddy/Caddyfile)

          SYNC_SCRIPT=$(cat <<EOF
          set -euo pipefail
          mkdir -p ${VM_APP_DIR}/caddy ${VM_APP_DIR}/import ${VM_APP_DIR}/backups
          echo '${COMPOSE_B64}' | base64 -d > ${VM_APP_DIR}/docker-compose.yml
          echo '${CADDY_B64}' | base64 -d > ${VM_APP_DIR}/caddy/Caddyfile
          echo '${ENV_B64}' | base64 -d > ${VM_APP_DIR}/.env
          chmod 600 ${VM_APP_DIR}/.env
          echo "sync-ok"
          EOF
          )
          output=$(az vm run-command invoke \
            --resource-group "${RESOURCE_GROUP}" \
            --name "${VM_NAME}" \
            --command-id RunShellScript \
            --scripts "${SYNC_SCRIPT}" \
            --query "value[0].message" -o tsv)
          echo "----- sync output -----"
          echo "${output}"
          echo "-----------------------"
          if ! echo "${output}" | grep -q "sync-ok"; then
            echo "Config sync failed — missing sync-ok sentinel."
            exit 1
          fi

          # Clean up the local env file so it isn't persisted in the runner.
          rm -f /tmp/env_body

      - name: Deploy to VM
        run: |
          NEW_IMAGE="${IMAGE_REPO}:${{ github.sha }}"
          DEPLOY_SCRIPT=$(cat <<EOF
          set -euo pipefail
          cd ${VM_APP_DIR}
          # Refresh ACR credentials (~3h TTL otherwise).
          az login --identity >/dev/null
          az acr login --name starkyacr >/dev/null
          docker compose pull
          docker compose up -d
          docker image prune -f --filter "until=168h" >/dev/null
          echo "deploy-ok image=${NEW_IMAGE}"
          EOF
          )
          # az vm run-command returns 0 even when the shell script inside the VM
          # exits non-zero — the inner exit code is in the response payload,
          # not bubbled up. Look for the "deploy-ok" sentinel and fail the step
          # explicitly if it's missing.
          output=$(az vm run-command invoke \
            --resource-group "${RESOURCE_GROUP}" \
            --name "${VM_NAME}" \
            --command-id RunShellScript \
            --scripts "${DEPLOY_SCRIPT}" \
            --query "value[0].message" -o tsv)
          echo "----- vm output -----"
          echo "${output}"
          echo "---------------------"
          if ! echo "${output}" | grep -q "deploy-ok image=${NEW_IMAGE}"; then
            echo "Deploy script did not complete successfully — missing deploy-ok sentinel."
            exit 1
          fi

      - name: Wait for Keycloak health
        run: |
          HEALTH_SCRIPT='docker inspect --format="{{.State.Health.Status}}" keycloak-server 2>/dev/null || echo missing'
          for attempt in $(seq 1 24); do
            status=$(az vm run-command invoke \
              --resource-group "${RESOURCE_GROUP}" \
              --name "${VM_NAME}" \
              --command-id RunShellScript \
              --scripts "${HEALTH_SCRIPT}" \
              --query "value[0].message" -o tsv | tr -d '[:space:]')
            echo "Attempt ${attempt}: container health = ${status}"
            if echo "${status}" | grep -q healthy; then
              echo "Keycloak is healthy."
              exit 0
            fi
            sleep 10
          done
          echo "Timeout: Keycloak did not report healthy within 240s."
          exit 1
```

- [ ] **Step 7.3: Watch the Actions run**

Open `https://github.com/afnan/starky-auth/actions` in a browser. The "Build and Deploy" workflow should have started within ~10s of the push.

Timings to expect:
- Checkout + Node setup: ~30s
- Tailwind build: ~20s
- Azure login: ~10s
- Docker build (first time, no cache): 2–4 min
- Docker push: 30–60s (~500 MB image)
- Deploy step (az vm run-command): ~30–60s
- Health wait: up to 240s (Keycloak JVM startup is slow; first boot also runs the schema migration)

Total: ~5–10 min on a cold run.

If any step fails, inspect the logs in the GitHub Actions UI. Common first-run failures:
- **Docker build fails on `COPY themes/starky/login/`**: Tailwind build didn't produce `styles.css` — check the "Verify built CSS exists" step output.
- **ACR push fails with 401**: OIDC login didn't acquire the AcrPush permission. Confirm `github-starky-frontend` has Contributor on `rg-starkyapp`, and the federated credential subject exactly matches `repo:afnan/starky-auth:ref:refs/heads/main`.
- **Sync step fails — missing sync-ok**: the `VM_ENV_FILE` secret is unset, truncated, or has CRLF line endings from Windows clipboard. Re-paste it using a plain text editor (LF line endings), save the secret again, and re-run.
- **Sync step fails — `/opt/starky-auth: No such directory`**: Task 6.1 didn't run cleanly. Re-run the bootstrap `mkdir -p` snippet.
- **Deploy step fails with "VM not found" or "Access denied"**: confirm `github-starky-frontend` has Virtual Machine Contributor (via Contributor-on-RG inheritance) and that you're in the right subscription.
- **Deploy fails pulling image from ACR**: the deploy script re-authenticates via `az login --identity` + `az acr login` on every run. If that step fails, the UAMI doesn't have `AcrPull` on `starkyacr`. Re-check role assignment in the portal.
- **Health wait times out**: Keycloak may still be starting. Browse to the VM in the portal → **Serial console** and run `docker compose logs keycloak` to see what's blocking.
